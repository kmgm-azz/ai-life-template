# 個人タスク管理リポジトリ運用ルール

日常のタスク、アイデア、参考情報を管理する個人用ワークスペース。

## 記録先

- 実行・進捗管理が必要なもの: GitHub Issues
- 日付単位の記録: `daily/`
- 継続して育てる内容: `ideas/`
- 繰り返し参照する情報: `references/`

## Issue管理

### 対象と事前確認

- 実行したいこと、後で取り組むこと、調べたいこと、残したいアイデアはIssueにする。
- 雑談や単純な質問はIssueにしない。
- 「Issueにして」「覚えておいて」「あとでやる」と明示された場合はIssueにする。
- 作成前に、同じ内容の未完了Issueがないか確認する。
- 締め切りが未提示なら「締め切りはいつですか？（未定でもOK）」と確認し、回答後に作成する。
- `未定`の場合は、Issue本文にも明記する。
- 内容によって目的や完了条件が大きく変わる場合だけ、必要最小限を質問する。

### 分類と本文

目的が明確なら分類を確認せず、最も近いテンプレートを使う。

- `daily`: 具体的なタスク、近いうちに行うこと
- `idea`: 育てたいアイデア、企画、試したいこと
- `reference`: 後で調査、整理、資料化する情報

タイトルは内容が短く分かるものにする。

本文には、分かる範囲で次を記載する。

- 背景・目的
- 締め切り
- やること
- 完了条件
- 参考リンクや補足

### ラベル

- 種別ラベルを1つ付ける: `daily`、`idea`、`reference`
- Areaラベルを1つ付ける。
  - 日常生活: `area: 生活`
  - 趣味: `area: 趣味`
  - 学習: `area: 学習`
  - 開発: `area: 開発`
- 明確な場合はトピックラベルも1つ付ける。
  - 例: `topic: ブログ`、`topic: 読書`、`topic: 運動`
- 必要なラベルがなければ、似たラベルとの重複を確認して作成する。
- 優先度など、Area・トピック以外のラベルは、明示または既存ルールがない限り設定しない。

### 作成・更新

- GitHub連携を優先し、必要に応じてGitHub CLIを使う。
- Issue作成後はProject `life` に追加し、Project項目を設定する。
- 進捗報告は既存Issueへ反映する。
- 対象のIssueが一意でない場合だけ確認する。
- 着手報告を受けたらProjectのStatusを`進行中`にする。
- 完了報告を受けたらIssueをクローズし、ProjectのStatusを`完了`にする。
- 作成・更新後は、タイトル、本文、ラベル、Project項目を確認する。
- Issue作成時は、Issue番号、タイトル、URLを報告する。

## GitHub Projects

- 未完了IssueはProject `life` で管理する。
- Issueを確認した際、Projectへの登録漏れがあれば補う。
- 既にクローズ済みの過去Issueは、指定がなければProjectへ追加しない。
- 追加時のStatusは`未着手`にする。
- `分類`はAreaラベルに合わせる。
- `トピック`はトピックラベルに合わせる。
- `期限`はIssue本文の締め切りに合わせる。未定なら空欄にする。
- 必要な選択肢がなければ、重複を確認して追加する。
- メインViewはボード形式とし、列をGitHub組み込みの`Status`、横グループを`分類`にする。

## Issueを使った提案

### 今日やること

未完了Issueから、次の順で3〜5件を提案する。

1. 明示された優先度
2. 期限の近さ、または期限超過
3. 現在着手できるか

各候補には次を含める。

- Issue番号
- タイトル
- 選定理由
- 今すぐできる最初の一歩

未設定の優先度を推測して変更しない。

## 記事・資料をMarkdownで残す場合

- Issueは構想・進捗管理に使い、Markdownファイルへ自動変換されるものとして扱わない。
- 内容が固まったら作業ブランチを作り、適切な保存先へ`.md`ファイルを作る。
- Pull Requestを作成し、本文へ`Closes #Issue番号`を記載する。
- 内容を確認してデフォルトブランチへマージする。
- マージ後、Issueがクローズされ、Projectが`完了`になっていることを確認する。
- 外部ブログへ直接投稿する場合は、公開URLをIssueへ記録してからクローズする。

## Git運用

- ファイル変更後は差分を確認し、同じ作業内でコミットして`origin`へpushする。
- 通常は`main`へ直接反映し、大きな変更やレビューが必要な場合は作業ブランチとPull Requestを使う。
- コミットは`<prefix>: <簡潔な日本語の要約>`とする。
  - `add`: 小さな追加
  - `change`: 小さな変更
  - `feat`: 利用者向け機能
  - `fix`: 不具合修正
  - `remove`: 削除
  - `docs`: 文書のみ
  - `chore`: 補助作業
- pushやマージに失敗した場合は、原因と残作業を報告する。

## セキュリティ

- APIキー、トークン、パスワード、個人情報をIssueやコミットへ記載しない。
- 会話に秘密情報が含まれていても転記しない。
- 公開される可能性がある内容に機微情報が含まれる場合は、ユーザーへ注意を促す。
- 外部公開前に、個人情報や非公開情報が含まれていないか確認する。
